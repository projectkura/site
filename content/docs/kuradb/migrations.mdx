---
title: Migrations
description: Generate and apply database migrations from your TypeScript schema.
---

KuraDB uses a schema-first migration workflow. You define tables in `src/schema.ts`, generate SQL migration files, review them, and apply them to your database — all from the txAdmin console.

## Workflow

<Steps>

<Step>

### Define your schema

Edit `src/schema.ts` to add or modify tables:

```ts
export const users = defineTable('public', 'users', {
  id: defineColumn('id', 'uuid', { primaryKey: true }),
  username: defineColumn('username', 'string'),
  createdAt: defineColumn('created_at', 'date', { defaultExpression: 'NOW()' }),
});
```

</Step>

<Step>

### Generate migrations

Run from the txAdmin console:

```
kuradb generate
```

This creates two files:
- `migrations/pending.sql` — The SQL to apply
- `lib/schema.generated.lua` — EmmyLua type definitions + `kura.db.tables` metadata

```
✓ Generated: migrations/pending.sql
✓ Generated Lua types: lib/schema.generated.lua
✓ Schema generated: 3 tables ready
Run "kuradb migrate" to apply changes
```

</Step>

<Step>

### Review the SQL

Open `migrations/pending.sql` and inspect the generated statements:

```sql
-- Migration v1: Create users table
CREATE TABLE IF NOT EXISTS "public"."users" (
  "id" UUID PRIMARY KEY,
  "username" VARCHAR(255) NOT NULL,
  "created_at" TIMESTAMP NOT NULL DEFAULT NOW()
);
```

<Callout type="warn" title="Always review">
  Especially in production — review the generated SQL before applying. You can edit `pending.sql` to add `ALTER TABLE` statements or adjust anything the generator doesn't cover.
</Callout>

</Step>

<Step>

### Apply migrations

```
kuradb migrate
```

```
Applying 3 migration(s)...
✓ v1: Create users table
✓ v2: Create posts table
✓ v3: Create comments table
✓ Updated Lua types
✓ Migration complete
```

If a migration fails, remaining migrations are skipped and the error is printed.

</Step>

</Steps>

## Generated files

### `migrations/pending.sql`

Plain SQL with all pending migrations. Safe to edit before running `kuradb migrate`.

```sql
-- Auto-generated migrations
-- Generated at: 2026-04-15T12:00:00.000Z

-- Migration v1: Create users table
CREATE TABLE IF NOT EXISTS "public"."users" (
  "id" UUID PRIMARY KEY,
  "username" VARCHAR(255) NOT NULL,
  "email" VARCHAR(255) NOT NULL,
  "password_hash" VARCHAR(255) NOT NULL,
  "created_at" TIMESTAMP NOT NULL,
  "updated_at" TIMESTAMP NOT NULL,
  "deleted_at" TIMESTAMP
);

-- Migration v2: Create posts table
CREATE TABLE IF NOT EXISTS "public"."posts" (
  "id" UUID PRIMARY KEY,
  "user_id" UUID NOT NULL,
  "title" VARCHAR(255) NOT NULL,
  "content" VARCHAR(255) NOT NULL,
  "published" BOOLEAN,
  "created_at" TIMESTAMP NOT NULL
);
```

### `lib/schema.generated.lua`

Auto-generated Lua metadata with EmmyLua annotations for IDE autocomplete:

```lua
-- Auto-generated by kuradb
-- Generated: 2026-04-15T12:00:00.000Z

---@class UsersRow
---@field id string
---@field username string
---@field email string
---@field passwordHash string
---@field createdAt number
---@field updatedAt number
---@field deletedAt number?

---@class PostsRow
---@field id string
---@field userId string
---@field title string
---@field content string
---@field published boolean?
---@field createdAt number
```

Use the generated types for autocomplete:

```lua
---@type UsersRow[]
local users = kura.db.raw.query.await('SELECT * FROM "public"."users"')
print(users[1].username) -- IDE autocomplete works
```

## Listening for migrations

Other resources can react when migrations are applied:

```lua
AddEventHandler('kuradb:schemaMigrated', function(schema)
  print('Schema updated: ' .. #schema.tables .. ' tables')
end)
```

## Tips

<Callout type="info">
- Always run `kuradb generate` before `kuradb migrate`
- `CREATE TABLE IF NOT EXISTS` is safe to re-run
- For column changes (add/remove/rename), edit `pending.sql` with `ALTER TABLE` statements
- Lua types are regenerated on both `generate` and `migrate`
- No resource restart is needed between generate and migrate
</Callout>
