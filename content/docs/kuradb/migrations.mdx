---
title: Migrations
description: Generate and apply database migrations from your TypeScript schema.
---

KuraDB uses a schema-first migration workflow. You define tables in `src/schema.ts`, generate named SQL migration files, review them, and apply them to your database from either the txAdmin console or the standalone CLI bundled in the release zip.

## Workflow

<Steps>

<Step>

### Define your schema

Edit `src/schema.ts` to add or modify tables:

```ts
export const users = defineTable('public', 'users', {
  id: defineColumn('id', 'uuid', { primaryKey: true }),
  username: defineColumn('username', 'string'),
  createdAt: defineColumn('created_at', 'date', { defaultExpression: 'NOW()' }),
});
```

</Step>

<Step>

### Generate migrations

Run from the txAdmin console or from the extracted `kuradb` release directory:

```
kuradb generate
```

Or give the migration a custom name:

```
kuradb generate add_users_table
```

If you only want to refresh `lib/schema.generated.lua` without creating a migration or running FiveM:

```
kuradb generate --types-only
```

This updates:
- `migrations/0000_<name>.sql` — The generated migration file
- `migrations/meta/_journal.json` — Migration journal
- `migrations/meta/0000_snapshot.json` — Schema snapshot for that generation
- `lib/schema.generated.lua` — EmmyLua type definitions + `kura.db.tables` metadata

```
✓ Generated migration: migrations/0000_init_users.sql
✓ Updated journal: migrations/meta/_journal.json
✓ Generated Lua types: lib/schema.generated.lua
✓ Schema generated: 3 tables ready
Run "kuradb migrate" to apply pending migration files
```

</Step>

<Step>

### Review the SQL

Open the generated migration file and inspect the statements:

```sql
-- Auto-generated by kuradb
-- Tag: init_users
-- Generated at: 2026-04-22T12:00:00.000Z

CREATE TABLE IF NOT EXISTS "public"."users" (
  "id" UUID PRIMARY KEY,
  "username" VARCHAR(255) NOT NULL,
  "created_at" TIMESTAMP NOT NULL DEFAULT NOW()
);
```

<Callout type="warn" title="Always review">
  Especially in production — review the generated SQL before applying. You can edit the generated `.sql` file to add `ALTER TABLE` statements or adjust anything the generator doesn't cover.
</Callout>

</Step>

<Step>

### Apply migrations

```
kuradb migrate
```

```
Applying 3 migration file(s)...
✓ Applied 0000_init_users.sql
✓ Applied 0001_add_posts.sql
✓ Applied 0002_add_comments.sql
✓ Updated Lua types
✓ Migration complete
```

If a migration fails, remaining migrations are skipped and the error is printed.

For the standalone CLI, set `KURADB_CONNECTION_STRING` first:

```bash
KURADB_CONNECTION_STRING="postgres://user:password@localhost:5432/fivem" kuradb migrate
```

</Step>

</Steps>

## Generated files

### `migrations/0000_<name>.sql`

Plain SQL for a single migration. Safe to edit before running `kuradb migrate`.

```sql
-- Auto-generated by kuradb
-- Tag: init_users_posts
-- Generated at: 2026-04-15T12:00:00.000Z

CREATE TABLE IF NOT EXISTS "public"."users" (
  "id" UUID PRIMARY KEY,
  "username" VARCHAR(255) NOT NULL,
  "email" VARCHAR(255) NOT NULL,
  "password_hash" VARCHAR(255) NOT NULL,
  "created_at" TIMESTAMP NOT NULL,
  "updated_at" TIMESTAMP NOT NULL,
  "deleted_at" TIMESTAMP
);

CREATE TABLE IF NOT EXISTS "public"."posts" (
  "id" UUID PRIMARY KEY,
  "user_id" UUID NOT NULL,
  "title" VARCHAR(255) NOT NULL,
  "content" VARCHAR(255) NOT NULL,
  "published" BOOLEAN,
  "created_at" TIMESTAMP NOT NULL
);
```

### `migrations/meta/_journal.json`

The migration journal tracks migration order and snapshot files:

```json
{
  "version": "1",
  "dialect": "postgresql",
  "entries": [
    {
      "idx": 0,
      "when": 1776859200000,
      "tag": "init_users",
      "filename": "0000_init_users.sql",
      "snapshot": "0000_snapshot.json"
    }
  ]
}
```

### `public.__kuradb_migrations`

Applied migrations are tracked in PostgreSQL so the state survives resource restarts:

```sql
SELECT "id", "tag", "applied_at"
FROM "public"."__kuradb_migrations"
ORDER BY "applied_at" ASC;
```

### `lib/schema.generated.lua`

Auto-generated Lua metadata with EmmyLua annotations for IDE autocomplete:

```lua
-- Auto-generated by kuradb

---@class UsersRow
---@field id string
---@field username string
---@field email string
---@field passwordHash string
---@field createdAt number
---@field updatedAt number
---@field deletedAt number?

---@class UsersInsert
---@field id string
---@field username string
---@field email string
---@field passwordHash string
---@field createdAt number
---@field updatedAt number
---@field deletedAt? number

---@class UsersUpdate
---@field username? string
---@field email? string
---@field passwordHash? string
---@field updatedAt? number
---@field deletedAt? number

---@class UsersTable : KuraDBTable<UsersRow, UsersInsert, UsersUpdate>
```

Use the generated types for autocomplete:

```lua
local users = kura.db.tables.users
local rows = kura.db.select():from(users):await()

print(rows[1].username) -- IDE autocomplete works
```

## Listening for migrations

Other resources can react when migrations are applied:

```lua
AddEventHandler('kuradb:schemaMigrated', function(schema)
  print('Schema updated: ' .. #schema.tables .. ' tables')
end)
```

## Tips

<Callout type="info">
- Always run `kuradb generate` before `kuradb migrate`
- Use `kuradb generate --types-only` when you only need fresh Lua types from the schema
- Migration files are applied by filename order from the journal
- Applied migrations are tracked in `public.__kuradb_migrations`
- If an already-applied migration file is changed on disk, KuraDB will stop and report it
- `CREATE TABLE IF NOT EXISTS` is safe to re-run for bootstrap-style migrations
- For column changes (add/remove/rename), edit the generated `.sql` file with `ALTER TABLE` statements
- Lua types are regenerated on both `generate` and `migrate`
- The release zip includes the CLI entrypoint, so type generation does not require a running FiveM server
- No resource restart is needed between generate and migrate
</Callout>
